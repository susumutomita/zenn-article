本書では、Zigで学習用のブロックチェインと簡易EVMを作りました。最後に、実用的なプロトコルへ近づけるときの作業を、現在のコードとの差から整理します。

この章は概念と調査の入口です。新しい実行可能コードは追加しません。仕様やプロジェクトの状況は変わるため、着手時にはリンク先の最新版とステータスを確認してください。

## EVM準拠へ進む

本書の`src/evm.zig`は、Solidityが生成する一部のバイトコードを観察するためのインタプリターです。EVM準拠を目指す場合は、命令に加えて次の規則を同じフォーク条件で実装します。

- 256ビット演算のすべての境界値とラップアラウンド。
- 命令ごとのガスと、メモリ拡張による追加ガス。
- アカウント、nonce、残高、コード、ストレージの状態遷移。
- CALL、DELEGATECALL、STATICCALL、CREATE、CREATE2の実行フレーム。
- ログ、例外、REVERTデータ、戻りデータ。
- 暗号ハッシュとプリコンパイル。
- フォークごとに追加・変更・廃止された命令とガス規則。

命令番号の入口には[Ethereum.orgのopcode表](https://ethereum.org/developers/docs/evm/opcodes/)を使えます。状態遷移とネットワークアップグレードを実装するときは、[Ethereum Execution Layer Specifications](https://github.com/ethereum/execution-specs)を基準にします。

### 次のチェックポイント

実装を進める場合は、完成版へ一度に混ぜず、次の順でチェックポイントを作ります。

| チェックポイント | 変更対象 | 受け入れテスト |
| --- | --- | --- |
| `ch16-sec01-u256` | `src/evm_types.zig` | 演算ごとの最大値、最小値、キャリー、ボロー |
| `ch16-sec02-gas` | `src/evm.zig` | 命令別ガス、メモリ拡張、OutOfGas |
| `ch16-sec03-accounts` | 新しい状態モジュール | nonce、残高、コード、ストレージの復元 |
| `ch16-sec04-call-frames` | `src/evm.zig` | ネスト呼び出し、REVERT、戻りデータ |
| `ch16-sec05-create` | EVMとブロックチェイン統合 | アドレス導出、initcode、runtime code |
| `ch16-sec06-conformance` | テストfixture | 公式仕様に由来する成功・失敗ケース |

## EVM Object Formatを追うときの注意

EVM Object Formatは、コードとデータの区分、事前検証、相対ジャンプなどを提案してきた一連のEIPです。ただし、[EOFv1のMeta EIPであるEIP-7692](https://eips.ethereum.org/EIPS/eip-7692)は、2026年7月の確認時点で`Stagnant`です。

したがって「次のEthereumで必ず採用される形式」として本書の実装へ先回りして組み込むのではなく、個別EIPのステータスと対象フォークを確認してから実装します。提案中の仕様と、メインネットで有効な仕様を同じ分岐へ混ぜてはいけません。

## コントラクト状態を永続化する

現在の実装はruntime codeを`contract_storage`とブロックへ保存します。一方、EVM実行コンテキストの`SSTORE`状態は、独立したコールをまたぐ正規状態として永続化されません。

次の段階では、少なくとも次の処理が必要です。

1. 実行前のアカウント状態を読み込む。
2. EVMが読み書きするストレージをトランザクション単位で隔離する。
3. 成功時だけ変更をコミットする。
4. REVERTやOutOfGasでは変更を破棄する。
5. ブロック内の全トランザクション後に状態コミットメントを求める。
6. 再起動と別ノード同期で同じ状態を復元する。

この作業を終えるまでは、`SSTORE`を含む一般的なSolidityコントラクトが永続状態を持つとは表現できません。

## rollupを理解する

rollupは、単にEVMを高速化した別チェインではありません。実行をL2側で行いながら、データや状態コミットメント、証明または異議申立てをL1へ結び付けるプロトコルです。

### Optimistic rollup

Optimistic rollupは、L2の状態更新をデフォルトでは正しいものとして扱い、異議申立て期間とfraud proofによって不正な遷移を争います。基礎は[Ethereum.orgのOptimistic rollup解説](https://ethereum.org/developers/docs/scaling/optimistic-rollups/)で確認できます。

本書のノードから進む場合、決定的な状態遷移、状態ルート、バッチ、L1へのデータ公開、異議申立てゲーム、期限、ボンドが追加で必要です。

### ZK-rollup

ZK-rollupは、L2で計算した状態遷移についてvalidity proofを作り、L1の検証コントラクトで確認します。基礎は[Ethereum.orgのZK-rollup解説](https://ethereum.org/developers/docs/scaling/zk-rollups/)で確認できます。

簡易EVMをzkEVMへ変えるには、実行トレースを出すだけでは足りません。各命令と状態アクセスを証明システムの制約へ落とし込み、公開入力、witness、状態コミットメント、証明生成、検証を一貫させる必要があります。

### 先に必要な共通基盤

どちらの方式でも、次の基盤は共通です。

- 再実行可能で決定的な状態遷移。
- トランザクションと状態のコミットメント。
- 履歴を復元できるデータ可用性。
- L1とL2のメッセージ形式。
- バッチの順序とファイナリティ。
- 不正または無効なバッチを拒否する検証規則。

現在の`syncChain`補助関数はhash、PoW、決定的genesis、連続index、親hash、重複を置換前に検証し、候補の選択基準は長さだけです。さらにP2P実通信からは未接続で、実際のノード同期は現在のtipへ直接つながる`BLOCK`のprefix追随だけです。このままrollupのフォーク選択やファイナリティへ流用できません。

## 性能最適化は準拠性の後に行う

EVM実装の高速化には、命令ディスパッチの改善、基本ブロックの事前解析、メモリ割り当ての削減、キャッシュ、JITやAOTなどの選択肢があります。しかし、誤った状態遷移を高速に計算しても意味はありません。

最適化は次の順で進めます。

1. 仕様から成功・失敗ケースを作る。
2. 参照実装と同じ結果になることを確認する。
3. 命令数、割り当て、メモリ量、実行時間を測る。
4. 1つの最適化だけを加える。
5. 全適合性テストを再実行する。
6. 同じ入力と同じ環境で再計測する。

`src/evm_debug.zig`のトレースは、どの命令で状態がずれたかを調べる入口として使えます。最適化後も、デバッグビルドでは観測可能性を残します。

## スマートコントラクトの安全性

VMが仕様どおりでも、その上で動くコントラクトが安全とは限りません。Solidityの公式ドキュメントは、再入可能性、ガス上限、外部呼び出し、`tx.origin`、整数境界などの注意点を[Security Considerations](https://docs.soliditylang.org/en/latest/security-considerations.html)にまとめています。

テストだけで表現しにくい不変条件には、[SMTCheckerと形式検証](https://docs.soliditylang.org/en/latest/smtchecker.html)も利用できます。ただし、検証結果は書いた仕様の範囲に依存します。誤った不変条件を証明しても、意図した安全性は得られません。

本書の`SimpleAdder.sol`はEVM経路を確認するfixtureであり、資産を預ける実用コントラクトの雛形ではありません。

## 次に実装する1つを選ぶ

完成版から先へ進むときは、次のどれか1つを選び、独立した節チェックポイントを作ります。

- フォーク選択: 累積workと確定済み履歴を考慮し、長さだけで候補を選ばない。
- 永続化: ブロックとコントラクト状態を保存し、再起動後に復元する。
- 署名: トランザクションの署名、送信者導出、アカウントnonce、リプレイ拒否を加える。
- EVM準拠: 1つの命令群とガス規則を公式仕様のテストと対にする。
- ネットワーク防御: 64 KiB超を明示拒否し、メッセージ別上限、タイムアウト、レート制限、認証を加える。
- PoS: 第15章の失敗ケースを満たす合意プロトコルを別ブランチで設計する。

変更後は`sh scripts/verify-book-code.sh`を実行し、本書で完成した範囲を壊していないことを確認します。
